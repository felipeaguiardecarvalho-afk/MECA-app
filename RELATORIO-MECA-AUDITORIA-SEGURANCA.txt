NOTA LEGAL
Este documento descreve o desenho de segurança tal como implementado no código referenciado.
Não substitui auditoria formal, teste de penetração em produção nem certificação.
Validar sempre a configuração real (Supabase Dashboard, Vercel, variáveis de ambiente).


1. ÂMBITO E OBJETIVO
--------------------------------------------------------------------------------
1.1 Âmbito
  - Aplicação web MECA (landing pública; rotas autenticadas e protegidas por middleware)
  - APIs Route Handlers (/api/*)
  - Supabase Auth + Postgres (RLS + RPC)
  - Dados pessoais: e-mail Auth; respostas ao questionário (12 itens Likert + metadados
    derivados — ver RELATORIO-MECA-PERGUNTAS-DIAGNOSTICO.txt); histórico e agregados
  - Papel privilegiado: master admin (e-mail em variável de servidor) com leitura
    global de diagnósticos e agregados via APIs servidor

1.2 Objetivo
  Avaliar superfícies de ataque, separação de funções, tratamento de dados (RGPD) e
  riscos do papel admin concentrado.


2. SUPERFÍCIE DE UTILIZADOR (UI) E DADOS
--------------------------------------------------------------------------------
2.1 Landing e navegação pública
  - A página inicial não apresenta dados de utilizadores nem métricas reais; apenas
    marketing e chamadas para login/diagnóstico.
  - Barra superior: marca + acesso a login (Entrar) ou, com sessão, ligações a Dashboard
    e Sair — sem exposição de PII na UI pública.

2.2 Questionário
  - O conteúdo das 12 perguntas é fixo no cliente (diagnostic-engine.ts); as respostas
    são enviadas apenas via POST /api/score após autenticação (e grant quando aplicável).
  - Do ponto de vista de privacidade, o conjunto de perguntas define o tipo de dado
    inferido (perfil organizacional/comportamental); a política de informação ao titular
    deve referir este tipo de tratamento.

2.3 Sessão obrigatória em APIs sensíveis
  - GET /api/benchmark exige sessão (não é recurso anónimo); utilizadores não autenticados
    não recebem benchmarks nem agregados pela API.


3. CLASSIFICAÇÃO DE DADOS
--------------------------------------------------------------------------------
  - Conta: e-mail (Auth)
  - Diagnósticos: respostas Likert, scores, arquétipo — potencialmente sensíveis
  - Agregados (admin): estatísticas calculadas no servidor; utilizadores normais não
    recebem, via benchmark, estatísticas sobre terceiros
  Classificação formal e base legal: responsável pelo tratamento / DPO


4. MODELO DE CONFIANÇA
--------------------------------------------------------------------------------
  [Browser] ←→ [Vercel / Next.js] ←→ [Supabase API + Postgres]

  Segredos só no servidor: SUPABASE_SERVICE_ROLE_KEY, MASTER_ADMIN_EMAIL
  NEXT_PUBLIC_* visíveis no browser (anon key pública por desenho; RLS + políticas API)


5. AUTENTICAÇÃO E SESSÃO
--------------------------------------------------------------------------------
5.1 Magic link (Supabase Auth), cookies SSR (@supabase/ssr)
5.2 /auth/callback: parâmetro "next" sanitizado (mitiga open redirect)
5.3 MFA: não implementado — avaliar risco

5.4 Verificações sugeridas
  [ ] Redirect URLs e Site URL no Supabase
  [ ] Revogação de sessão se conta admin comprometida


6. AUTORIZAÇÃO — UTILIZADOR VS MASTER ADMIN
--------------------------------------------------------------------------------
6.1 Identidade
  - Decisões de autorização com base em supabase.auth.getUser() / middleware. Nunca
    confiar em e-mail ou user_id enviados pelo cliente no corpo do pedido.

6.2 Master admin
  - MASTER_ADMIN_EMAIL vs e-mail da sessão (isAdmin, normalizado)
  - Middleware: sem exigência de access_grants para /assessment e /dashboard
  - POST /api/score: sem grant para admin; com grant para utilizador normal
  - GET /api/user/history: RLS para utilizador; admin + service role para todas as linhas;
    503 se faltar service role (evita leitura parcial insegura)
  - GET /api/benchmark: sessão; utilizador — referência + texto aggregated; admin —
    globalInsights via service role

6.3 Risco single super-admin
  - Um e-mail com elevado poder de leitura; sem RBAC fino na base de dados.
  - Mitigação operacional: MFA, segredo das variáveis, roadmap para roles em BD/Auth.

6.4 Residual
  - Comprometer conta admin ou variáveis Vercel compromete confidencialidade global.


7. ROW LEVEL SECURITY E RPC
--------------------------------------------------------------------------------
7.1 access_codes: bloqueio a cliente; gestão fora da app ou RPC
7.2 responses: SELECT/INSERT com auth.uid() = user_id; sem UPDATE/DELETE na migração base
7.3 access_grants: SELECT/INSERT próprio utilizador
7.4 validate_access_code: SECURITY DEFINER, search_path = public
7.5 responses.is_admin: flag informativa; não substitui controlo de sessão

7.6 Auditoria SQL
  [ ] Políticas RLS em produção alinhadas às migrações
  [ ] Permissões EXECUTE na RPC


8. CHAVE SERVICE ROLE
--------------------------------------------------------------------------------
8.1 Uso: insert opcional; leituras cross-tenant só após isAdmin em rotas definidas
8.2 Risco: vazamento = acesso total API Supabase
8.3 Verificações: só variáveis servidor; não em repositório; rotação após incidente


9. APIs — SUPERFÍCIE E DADOS
--------------------------------------------------------------------------------
  POST /api/access-code — RPC; sessão obrigatória
  POST /api/score — answers apenas; user_id no servidor
  GET /api/user/history — viewer.role (metadado UI; não autorização por si)
  GET /api/benchmark — sessão obrigatória

  Lacunas: rate limiting, validação de tamanho de payload, idempotência

RESUMO DA PÁGINA MECA
=====================

Atualizado: 21 de abril de 2026

Resumo funcional e técnico consolidado:
  docs/RELATORIO_SITUACAO_ATUAL.md

Segurança / auditoria:
  docs/RELATORIO_AUDITORIA_SEGURANCA.md — críticos endereçados; revisão 20 abr 2026
  (env fail-fast, modo dev bloqueado em produção, TRUST_PROXY, login-intent neutro,
  logs com PII redigida, limites IA globais). Riscos residuais em §6–8.

1) Stack e arquitetura
- Frontend: Next.js (App Router) + React + TypeScript.
- Auth e dados: Supabase (Auth + tabelas como responses e access_grants).
- Navegação principal:
  - / = landing page
  - /login = autenticação por e-mail (magic link)
  - /assessment = questionario/diagnostico
  - /dashboard = resultado, historico e comparacao
  - /access-code = ponte de grant/acesso

2) Regras de acesso
- Rotas publicas: /, /login, /diagnostico e callback de autenticacao.
- Rotas protegidas por sessao: /assessment, /dashboard, /plano-de-acao, /access-code.
- Se o usuario nao estiver autenticado e tentar acessar rota protegida, vai para /login?next=...
- O middleware verifica se existe access_grant para permitir o fluxo correto.
- Se o usuario ja concluiu o diagnostico e nao pode refaze-lo, /assessment redireciona para /dashboard.

3) Fluxo de login (canonico)
- O usuario informa o e-mail em /login.
- O frontend chama POST /api/auth/magic-link.
- O servidor usa auth.admin.generateLink, envia o link por e-mail e responde apenas
  { sent: true } (sem action_link/token na resposta HTTP).
- O usuario abre o link → /auth/confirm (hash) ou fluxo com /auth/callback (codigo PKCE)
  conforme configuracao Supabase.

4) Rate limit e e-mail
- Rotas de auth e magic link tem rate limit aplicado no servidor.
- O caminho generateLink + SMTP/Resend reduz dependencia do rate limit publico do Auth.

5) Fluxo do diagnostico
- O usuario responde perguntas em escala de 1 a 5.
- Existe timer por pergunta.
- Se o tempo acabar, a resposta neutra pode ser aplicada automaticamente.
- No final, o frontend envia as respostas para POST /api/score.
- A API calcula/persiste o diagnostico e devolve o id do resultado.
- O frontend redireciona para /dashboard?saved=<id>.

6) Fluxo do dashboard
- Carrega historico via /api/user/history.
- Usuario comum: ve somente os proprios diagnosticos.
- Master/admin: pode ver dados mais amplos e insights globais.
- O usuario pode:
  - escolher qual diagnostico visualizar
  - comparar com outro diagnostico do proprio historico
  - exportar PDF (admin/MFA conforme rota)
  - sair da sessao

7) Regras de negocio atuais
- Login por e-mail.
- Depois de concluir o diagnostico, o acesso principal passa a ser o dashboard.
- O refazer diagnostico nao e livre para todos; depende das regras em access_grants.
- Existe um master unico definido pela regra de negocio atual.

8) Dados envolvidos
- responses: guarda os diagnosticos concluidos e scores.
- access_grants: controla permissao de acesso e possibilidade de refazer diagnostico.

9) Pontos para discutir com o engenheiro de aplicacoes
- Politica de TRUST_PROXY e observabilidade de rate limit em producao.
- Superficie LLM (premium) e custos/guardrails adicionais.
- Deliverability (Resend/domínio FROM) e URLs de redirect no Supabase.

10) Estado atual
- Login funcional.
- Dashboard funcional.
- Historico de diagnosticos visivel.
- Comparacao entre diagnosticos implementada.
- Controle de acesso por sessao e grant implementado.

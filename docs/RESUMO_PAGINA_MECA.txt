RESUMO DA PÁGINA MECA
=====================

Atualizado: 22 de abril de 2026

Resumo funcional e técnico consolidado:
  docs/RELATORIO_SITUACAO_ATUAL.md

Segurança / auditoria:
  docs/RELATORIO_AUDITORIA_SEGURANCA.md — críticos endereçados; revisão 20 abr 2026
  (env fail-fast, modo dev bloqueado em produção, TRUST_PROXY, login-intent neutro,
  logs com PII redigida, limites IA globais). Riscos residuais em §6–8.

1) Stack e arquitetura
- Frontend: Next.js (App Router) + React + TypeScript.
- Auth e dados: Supabase (Auth + tabelas como profiles, responses e access_grants).
- Navegação principal:
  - / = landing page
  - /login = autenticação por e-mail (magic link)
  - /onboarding = cadastro mínimo (nome) antes do primeiro diagnóstico
  - /assessment = questionário/diagnóstico
  - /dashboard = resultado, histórico e comparação

2) Regras de acesso
- Rotas públicas: /, /login, /diagnostico, /fundamentos, /arquetipos, callback de autenticação.
- Rotas protegidas por sessão: /assessment, /dashboard, /plano-de-acao, /onboarding (redireciona se sem sessão).
- Se o utilizador não estiver autenticado e tentar aceder rota protegida, vai para /login?next=...
- O middleware exige sessão em REQUIRES_SESSION; não exige “código da organização”.
- Se o utilizador já concluiu o diagnóstico e não tem can_take_diagnostic, /assessment redireciona para /dashboard.

3) Fluxo de login (canónico)
- O utilizador informa o e-mail em /login.
- O frontend chama POST /api/auth/magic-link.
- O servidor usa auth.admin.generateLink, envia o link por e-mail e responde apenas
  { sent: true } (sem action_link/token na resposta HTTP).
- O utilizador abre o link → /auth/confirm (hash) ou fluxo com /auth/callback (código PKCE)
  conforme configuração Supabase.

4) Rate limit e e-mail
- Rotas de auth e magic link têm rate limit aplicado no servidor.
- O caminho generateLink + SMTP/Resend reduz dependência do rate limit público do Auth.

5) Fluxo do diagnóstico
- Primeiro acesso: sem full_name em profiles → /onboarding?next=/assessment.
- O utilizador responde perguntas em escala de 1 a 5.
- Existe timer por pergunta.
- Se o tempo acabar, a resposta neutra pode ser aplicada automaticamente.
- No final, o frontend envia as respostas para POST /api/score.
- A API calcula/persiste o diagnóstico e bloqueia novo diagnóstico (access_grants) até unlock admin.
- O frontend redireciona para /dashboard?saved=<id>.

6) Fluxo do dashboard
- Carrega histórico via /api/user/history.
- Utilizador comum: vê somente os próprios diagnósticos.
- Master/admin: pode ver dados mais amplos e insights globais.
- O utilizador pode:
  - escolher qual diagnóstico visualizar
  - comparar com outro diagnóstico do próprio histórico
  - exportar PDF (admin/MFA conforme rota)
  - sair da sessão

7) Regras de negócio atuais
- Login por e-mail.
- Depois de concluir o diagnóstico, o acesso principal passa a ser o dashboard.
- Refazer diagnóstico só após POST /api/admin/unlock-diagnostic (master + MFA).
- Existe um master único definido pela regra de negócio atual.

8) Dados envolvidos
- profiles: nome, profissão, telefone (onboarding).
- responses: guarda os diagnósticos concluídos e scores.
- access_grants: controla permissão e possibilidade de novo diagnóstico.

9) Pontos para discutir com o engenheiro de aplicações
- Política de TRUST_PROXY e observabilidade de rate limit em produção.
- Superfície LLM (premium) e custos/guardrails adicionais.
- Deliverability (Resend/domínio FROM) e URLs de redirect no Supabase.

10) Estado atual
- Login funcional.
- Dashboard funcional.
- Histórico de diagnósticos visível.
- Comparação entre diagnósticos implementada.
- Controlo de acesso por sessão e can_take_diagnostic implementado.

11) Relatório PDF (atualização 21 abr 2026)
- Geração HTML/PDF para admin via GET /api/report/generate (requireAdminWithMfa).
- Layout premium: capa, hero do arquétipo, gráfico do quadrante MECA,
  sumário executivo em 3 blocos (Momento/Risco/Alavanca), barras por pilar,
  gráfico radar SVG dos 4 pilares (Mentalidade/Engajamento/Cultura/Performance),
  banner de identidade do arquétipo, plano de ação em checklist, teorias ao final.
- Motor de paginação block-based: cada secção usa break-before:page, blocos atómicos
  com break-inside:avoid, títulos com break-after:avoid-page. Corrige truncagem
  silenciosa que existia com overflow:hidden + min-height em .page.
- Detalhe em docs/RELATORIO_ENGENHARIA_SOFTWARE.md §7.6.

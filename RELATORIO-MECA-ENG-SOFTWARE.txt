1. SUMÁRIO EXECUTIVO
--------------------------------------------------------------------------------
O MECA é uma aplicação web multiutilizador (SaaS) para diagnóstico organizacional,
com autenticação Supabase (magic link), controlo de acesso por código, armazenamento
de respostas, painel com histórico, benchmarks, evolução temporal, radar (6 dimensões)
e exportação PDF.

A app inclui autorização em dois níveis: utilizador normal (RLS + código de acesso) e
master admin (MASTER_ADMIN_EMAIL + SUPABASE_SERVICE_ROLE_KEY para leituras globais).

A interface segue padrão de produto SaaS premium: tipografia Inter, fundo branco,
contentores centrados (max-width 1100px), botões primários pretos e secundários em
outline; barra superior minimalista (marca MECA + Entrar, ou Dashboard + Sair quando
há sessão). Lógica de negócio inalterada face a versões anteriores baseadas em outra
identidade visual.

Deploy alvo: Vercel + domínio. Lógica sensível em /app/api/*; Postgres com RLS e RPC
validate_access_code.

Segurança detalhada: RELATORIO-MECA-AUDITORIA-SEGURANCA.txt
Inventário de perguntas do questionário: RELATORIO-MECA-PERGUNTAS-DIAGNOSTICO.txt


2. OBJETIVO DO PRODUTO (ESCOPO FUNCIONAL)
--------------------------------------------------------------------------------
2.1 Funcionalidades presentes
  - Landing (/) — hero, CTAs, secções em grelha com cards (como funciona, descobertas,
    público-alvo, CTA final)
  - Login magic link (/login), callback (/auth/callback)
  - Código de acesso (/access-code) — RPC validate_access_code
  - Questionário (/assessment) — 12 perguntas Likert 1–5 (ver §3.6)
  - Dashboard: radar, dimensões, benchmark, evolução, histórico, PDF (/dashboard)
  - Modo master admin: histórico global, insights agregados, tendências por utilizador
  - APIs: /api/score, /api/access-code, /api/benchmark, /api/user/history
  - Middleware: sessão; grant obrigatório exceto master admin nas rotas protegidas
  - Redirecionamentos: /access → /access-code; /results → /dashboard; /reports → /dashboard

2.2 Fora do escopo ou parcial
  - Back-office na app para emitir códigos (Supabase Studio / SQL / seed)
  - RBAC na base de dados (vários admins) — hoje um MASTER_ADMIN_EMAIL
  - i18n completa; E2E; APM; PDF server-side institucional


3. ARQUITETURA TÉCNICA
--------------------------------------------------------------------------------
3.1 Frontend
  - Next.js 15 (App Router), React 19, Tailwind 3
  - Fonte: Inter (next/font/google) em layout.tsx — corpo e títulos; themeColor #ffffff
  - Layout global: AppNav fixo + main; páginas com px-6 py-16 onde aplicável; landing e
    dashboard alinhados a max-w-[1100px] (ou max-w-4xl no hero da home)
  - Componentes cliente: login, access-code, assessment, dashboard (DashboardClient),
    AuthNav, charts (RadarChart, TrendLineChart — traços neutros cinza/preto)
  - Ficheiros UI principais: src/components/landing/HomeLanding.tsx, AppNav.tsx,
    AuthNav.tsx; app/login/LoginForm.tsx, access-code/AccessForm.tsx; icon.tsx (favicon M
    em #171717)

3.2 Backend (BFF Next)
  - Route Handlers em src/app/api/*/route.ts
  - Sessão Supabase: cookies (@supabase/ssr)
  - Motor: lib/diagnostic-engine.ts — POST /api/score recalcula no servidor
  - Admin: src/lib/auth/isAdmin.ts — e-mail da sessão vs MASTER_ADMIN_EMAIL (só servidor)
  - Service role: lib/supabase/admin.ts — cross-tenant após isAdmin; 503 se faltar chave
  - Clientes Supabase sem genérico Database estrito (rpc / PostgREST)

3.3 Autorização (resumo)
  - Utilizador: RLS; access_grants para /assessment e /dashboard (middleware)
  - Master admin: middleware sem exigência de grant; POST /api/score sem grant; GET
    history/benchmark com service role para dados globais
  - responses.user_id da sessão; is_admin no insert quando aplicável (migração 0002)

3.4 Dados (Supabase / Postgres)
  - 0001_meca_full_schema.sql — schema core + RPC
  - 0002_responses_is_admin.sql — responses.is_admin boolean default false

3.5 Ficheiros-chave
  - src/lib/benchmark-insights.ts — agregados globais (admin)
  - src/types/database.ts — contrato de schema
  - supabase/config.toml, seed.sql; npm: db:push, db:reset, clean

3.6 Conteúdo do questionário (referência)
  - 12 perguntas em MECA_QUESTIONS (diagnostic-engine.ts): 6 dimensões × 2 itens
    (mentalidade, engajamento, cultura, performance, direction, capacity).
  - Listagem textual completa: RELATORIO-MECA-PERGUNTAS-DIAGNOSTICO.txt

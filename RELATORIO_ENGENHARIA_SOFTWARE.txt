--------------------------------------------------------------------------------
3. FLUXO DE ENTRADA (INCLUI PÁGINA DE LOGIN)
--------------------------------------------------------------------------------

1) Rotas protegidas sem sessão → /login?next=...
2) /login: OTP por e-mail; link mágico → /auth/callback
3) Callback define sessão e redireciona (ex. /access-code)
4) access_grants via código; can_take_diagnostic controla novo /assessment
5) AuthNav e HomeLandingActions usam /api/user/access-state para CTAs

A rota /login é parte integral do produto com autenticação ativa.

--------------------------------------------------------------------------------
4. DASHBOARD — DADOS
--------------------------------------------------------------------------------

Bootstrap sessionStorage, ?saved=, GET /api/user/history. getArchetype no cartão.

--------------------------------------------------------------------------------
5. PLANO DE AÇÃO
--------------------------------------------------------------------------------

getActionPlan por pilar mais baixo. Protegido no middleware.

--------------------------------------------------------------------------------
6. ESTILOS, BUILD, TESTES
--------------------------------------------------------------------------------

Tailwind + ds-*; npm run clean para cache .next; testes: diagnostic-engine, action-plan,
styling-system, access-control.

--------------------------------------------------------------------------------
7. RESUMO EXECUTIVO
--------------------------------------------------------------------------------

App com /login (OTP Supabase), callback, access-code, um diagnóstico por utilizador
(can_take_diagnostic) e unlock admin. Plano de ação por action-plan.ts. Middleware
protege /assessment, /dashboard, /plano-de-acao; não autenticados → /login.
